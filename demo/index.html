Reactive keys, automatic aborts

02 ·

Optimistic mutations, with rollback

The classic optimistic UI pattern: write to the cache before the network responds. If the server says yes, the optimistic write is canonical. If it says no, onError restores the snapshot captured by onMutate — visible rollback, no UI flicker.

    Mutation status: idle
    What just happened: Adding a todo shows it instantly with an optimistic badge. On server success (1s delay), the optimistic flag drops and the list reflects server truth. With the failure toggle on, the row appears, then rolls back to its pre-mutation state. onSettled always fires either way — your spinner cleanup is safe.

    Cross-tab invalidation

    Invalidation across tabs is even more useful than setData propagation: small payload (just the key), and every observing tab triggers a fresh refetch. The pattern for "user mutated something on server X" — broadcast invalidate, every tab pulls latest.

    What just happened: The "broadcasts to all tabs" button calls qc.invalidate(['server-time']). Locally the entry refetches. Across tabs, the receiving client runs the same invalidate (with processingRemote set), which also refetches — but doesn't re-broadcast. Result: every tab's clock updates in lockstep with one click.

    Cross-tab fetch deduplication — five tabs, one request

    The feature no other query library ships. When many tabs poll the same endpoint, only the leader hits the network; followers receive the result over BroadcastChannel. Open this in two tabs, make one the leader and one a follower, then hit Refresh in either — watch the follower's network counter stay at zero.

    How to demo this: Open a second tab. Leave this one as LEADER, switch the other to FOLLOWER. Click Refresh in the follower — its network counter stays 0 while the leader's climbs, and both tabs show the same fresh data. The follower broadcast a fetch-req; the leader fetched once and broadcast the result. Close the leader tab and the follower self-fetches after a 3s fallback — no request is ever lost.

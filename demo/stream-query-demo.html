Streaming queries — chunks in, reactive UI out

A chunked source — SSE frames, websocket messages, a paginated cursor — pumped through lite-stream into the same lite-query cache a fetch query uses, driving the UI at 60 fps with zero extra signals allocated and nothing concatenated on the hot path. The latest value lands in the entry's own data signal; the entry stays monomorphic; teardown closes the connection.

Latest mode — a 60 fps signal feed

The source yields one sample per animation frame. In latest mode each frame is a single write into the entry's data signal — the transform returns the raw value, allocating nothing. The trace below is plotted from a pre-allocated Float64Array ring; watch the heap readout stay flat while it runs.

Switching tabs unmounts this scene's reader. With no observers left, the stream is aborted (iterator.return()) and the entry is GC-scheduled — lazy in, clean out.

Buffer mode — a bounded sliding window

The source emits log events. In buffer mode the entry holds a window of the most recent maxBuffer values; once it fills, each new event drops the oldest and bumps droppedCount. Memory is bounded by the window, not by stream length.

Lifecycle — detach closes it, invalidate restarts it

A stream is lazy: nothing connects until a reader subscribes. The last reader leaving aborts the iterator and resets the entry to idle; invalidate aborts the live connection and re-establishes a fresh one (the count resets).
